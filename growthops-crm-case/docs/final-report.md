# GrowthOps CRM Case Project 최종 해설 보고서

## 1. 프로젝트 개요

이 프로젝트는 마케팅 캠페인, 리드, 행동 이벤트, 전환 데이터가 함께 존재하는 CRM 상황에서 영업 후속 대응 우선순위를 계산하는 문제를 다룬다. 단순히 리드 목록을 정리하는 것이 아니라, 데이터 품질 문제를 먼저 확인하고 신뢰 가능한 리드만 자동 점수 계산 대상으로 분리하는 데 초점을 둔다.

이번 구현은 완성형 SaaS 제품이 아니라 Python과 CSV를 중심으로 동작하는 CRM 데이터 파이프라인 MVP다. 핵심 구현은 더미데이터 생성부터 최종 CSV 산출까지 한 번에 실행되는 구조이며, FastAPI는 산출된 CSV를 조회하기 위한 선택 기능으로만 추가했다. React, Airflow, 클라우드 저장소 같은 운영 구성은 포함하지 않았다.

최종 산출물은 마케팅 담당자와 세일즈 담당자가 다음 액션을 판단하는 데 사용할 수 있다. 예를 들어 Hot Lead는 빠른 직접 연락 대상으로 보고, Warm Lead는 nurture 이메일 후 재확인 대상으로 보며, 데이터 품질 문제가 있는 리드는 Manual Review로 분리해 먼저 확인하도록 한다.

## 2. 문제 상황 분석

실제 CRM 환경에서는 캠페인 정보, 리드 정보, 고객 행동 이벤트, 전환 데이터가 서로 다른 출처에 흩어져 있는 경우가 많다. 광고 플랫폼, 이메일 마케팅 도구, CRM, 웹 로그, 결제 또는 계약 데이터가 각각 다른 형식으로 관리되면 전체 고객 여정을 한 번에 보기 어렵다.

Excel로 수작업 취합을 계속할 수도 있지만, 데이터가 늘어날수록 반복 작업이 많아지고 실수가 발생하기 쉽다. 또한 리드 중복, 이메일 누락, 잘못된 캠페인 참조, UTM 누락, 이벤트 중복과 같은 문제가 숨어 있으면 단순 합계나 필터만으로는 정확한 판단이 어렵다.

특히 클릭 수나 이메일 오픈 수만 보고 관심도를 판단하면 실제 전환 가능성이 왜곡될 수 있다. 짧은 시간에 비정상적으로 많은 이벤트가 발생한 리드는 자동화된 트래픽일 수 있고, 중복 리드는 캠페인 성과를 과대 계산할 수 있다. 따라서 행동량뿐 아니라 데이터 품질과 전환 신호를 함께 봐야 한다.

## 3. 문제 정의

먼저 여러 CSV 데이터를 분석 가능한 구조로 정제해야 한다. 날짜, 숫자, 문자열 공백, 이메일 형식처럼 기본적인 표준화가 되어야 이후 품질 검증과 점수 계산이 안정적으로 동작한다.

다음으로 데이터 품질 문제를 탐지해야 한다. 이메일이 없는 리드, 존재하지 않는 캠페인을 참조하는 리드, 존재하지 않는 리드를 참조하는 이벤트나 전환 데이터, 중복 이메일, 봇성 행동 의심 리드는 자동 판단에 사용하기 어렵다.

또한 자동 점수 계산 대상과 수동 검토 대상을 분리해야 한다. 모든 데이터를 억지로 점수화하면 잘못된 리드가 Hot Lead로 분류될 수 있기 때문이다.

마지막으로 리드별 우선순위와 캠페인별 성과를 재현 가능한 결과물로 산출해야 한다. 같은 입력을 넣으면 같은 결과 CSV가 만들어져야 검증과 설명이 가능하다.

## 4. 해결 선택지 비교

### 선택지 A: Excel 수작업 리포트 유지

장점은 초기 비용이 낮고 별도 개발 없이 바로 시작할 수 있다는 점이다. 소량 데이터에서는 담당자가 직접 확인하면서 유연하게 수정할 수도 있다.

단점은 반복 작업이 많고, 데이터가 늘어날수록 실수가 발생하기 쉽다는 점이다. 품질 검증 규칙을 일관되게 적용하기 어렵고, 같은 입력에 대해 항상 같은 결과가 나온다고 보장하기도 어렵다.

### 선택지 B: 머신러닝 기반 리드 스코어링

장점은 충분한 학습 데이터가 있을 경우 복잡한 패턴을 반영할 수 있다는 점이다. 사람의 단순 규칙으로 찾기 어려운 전환 가능성 신호를 모델이 학습할 수도 있다.

단점은 현재 과제 범위에 비해 준비해야 할 것이 많다는 점이다. 학습용 라벨, 검증 데이터, 성능 지표, 모델 해석, 운영 모니터링이 필요하다. 더미데이터 기반의 작은 MVP에서는 모델 결과가 오히려 설명하기 어려울 수 있다.

### 선택지 C: CSV 기반 ETL 파이프라인 + 규칙 기반 스코어링

장점은 구현 범위가 명확하고 결과를 설명하기 쉽다는 점이다. 품질 검증, Manual Review 분리, 점수 계산 근거, 캠페인 지표 산출 과정을 코드와 CSV로 확인할 수 있다. 작은 데이터셋에서도 재현 가능하게 동작한다.

단점은 규칙과 가중치가 사람이 정한 기준에 의존한다는 점이다. 실제 운영에서는 영업 결과 데이터를 기반으로 가중치가 적절한지 검증하고 조정해야 한다.

## 5. 최종 선택

이번 MVP에서는 선택지 C인 CSV 기반 ETL 파이프라인과 규칙 기반 스코어링을 선택했다.

이 방식은 과제 범위에 적합하다. 완성형 서비스보다 데이터 생성, 정제, 검증, 점수 계산, 지표 산출의 흐름을 끝까지 보여주는 것이 중요하기 때문이다.

또한 결과가 설명 가능하다. 각 리드가 왜 Hot, Warm, Cold, Manual Review로 분류되었는지 `reason` 컬럼에 한국어 문장으로 남긴다.

데이터 품질 문제도 명시적으로 분리할 수 있다. 이메일 누락, 중복 이메일, 잘못된 캠페인 참조, 봇성 행동 의심 리드는 자동 점수 계산에서 제외한다.

작은 데이터셋에서도 재현 가능하다는 점도 중요하다. 난수나 외부 API를 사용하지 않고, 기준일도 고정해 같은 입력에 대해 같은 결과를 만들 수 있다.

마지막으로 신입 과제 수준에서 코드와 지표의 의미를 설명하기 좋다. 각 모듈의 역할이 분리되어 있고, 점수 계산 규칙도 비즈니스 의미와 연결되어 있다.

## 6. 구현 구조

- `data_generator.py`: 캠페인, 리드, 이벤트, 전환 더미데이터를 생성하고 원천 CSV로 저장한다.
- `loader.py`: `data/raw/`에 저장된 CSV 파일을 pandas DataFrame으로 로딩한다.
- `cleaner.py`: 문자열 공백 제거, 이메일 소문자화, 날짜 변환, 숫자 변환 등 기본 정제와 표준화를 수행한다.
- `quality_validator.py`: 데이터 품질 이슈를 탐지하고 severity를 부여한다. Manual Review 대상 리드도 이 단계에서 식별한다.
- `scorer.py`: Manual Review가 아닌 리드에 대해 규칙 기반 점수를 계산하고 등급, 사유, 추천 액션을 생성한다.
- `metrics.py`: 캠페인별 리드 수, 유효 리드 수, Hot 리드 수, 전환 수, 전환율, 품질 이슈 수를 계산한다.
- `reporter.py`: 최종 CSV 3개를 저장하고 콘솔 요약을 출력한다.
- `main.py`: 전체 실행 흐름을 순서대로 호출하는 진입점이다.

## 7. ETL/ELT 파이프라인 흐름

파이프라인은 먼저 더미 Raw 데이터를 생성한다. 캠페인, 리드, 이벤트, 전환 데이터가 각각 CSV로 만들어지며, 일부 품질 이슈가 의도적으로 포함된다.

생성된 Raw CSV는 `data/raw/`에 저장된다. 이후 로더가 CSV를 읽고, 정제 모듈이 날짜와 숫자 타입을 표준화하고 문자열 공백을 제거한다.

정제된 데이터는 품질 검증 단계로 전달된다. 이 단계에서 누락, 중복, 잘못된 참조, 봇성 행동 의심 같은 이슈가 탐지된다.

그 다음 리드 스코어링을 수행한다. Manual Review 대상은 자동 계산에서 제외하고, 정상 리드는 행동 이벤트, 전환 이벤트, 회사 정보, 캠페인 정보를 반영해 점수를 계산한다.

마지막으로 캠페인 지표를 계산하고 `data/processed/`에 최종 결과 CSV를 저장한다. 산출물은 `lead_scores.csv`, `campaign_metrics.csv`, `data_quality_report.csv` 세 가지다.

## 8. 데이터 품질 처리 방식

데이터 품질 이슈는 HIGH, MEDIUM, LOW로 나눈다. HIGH는 자동 스코어링이나 캠페인 귀속을 직접 왜곡할 수 있는 문제다. MEDIUM은 반드시 오류라고 단정할 수는 없지만 성과 해석에 영향을 줄 수 있는 문제다. LOW는 보조 정보 누락처럼 핵심 식별자는 유지되지만 점수 계산 근거가 줄어드는 문제다.

Manual Review 기준은 이메일 누락, 동일 이메일 중복, 유효하지 않은 `source_campaign_id`, 봇성 행동 의심이다. 이 조건 중 하나라도 해당하면 자동 점수 계산에서 제외한다.

자동 계산에서 제외한 이유는 잘못된 데이터를 점수화하면 영업 우선순위가 왜곡될 수 있기 때문이다. 예를 들어 이메일이 없는 리드는 후속 연락이 어렵고, 중복 이메일은 같은 사람에게 중복 연락할 위험이 있다. 봇성 행동은 실제 관심 신호가 아닐 수 있어 높은 행동 점수를 주기 어렵다.

품질 이슈는 캠페인 성과 해석에도 영향을 준다. 중복 리드는 리드 수를 부풀리고, 잘못된 캠페인 참조는 성과 귀속을 어렵게 만든다. UTM 누락은 유입 경로 추적을 약하게 만들고, 잘못된 전환 참조는 전환율 해석을 조심스럽게 만든다.

## 9. 리드 우선순위 알고리즘

리드 점수는 규칙 기반으로 계산한다. Manual Review가 아닌 정상 리드는 기본 10점에서 시작하고, 행동 이벤트, 전환 이벤트, 회사/직무/산업군 정보, 캠페인 정보를 반영해 점수를 더하거나 뺀다. 최종 점수는 0점에서 100점 사이로 제한한다.

행동 이벤트 점수는 관심도 강도에 따라 다르게 부여한다. 이메일 오픈은 약한 관심 신호로 +5점, 링크 클릭과 랜딩 방문은 더 적극적인 행동으로 각각 +10점, 웨비나 신청은 +20점, 데모 요청은 직접적인 구매 검토 신호로 +30점을 준다. 최근 30일 이벤트는 1건당 +2점, 최대 +20점까지 반영한다.

전환 이벤트는 비즈니스 결과에 더 가까운 신호다. 상담 요청은 +25점, 계약은 +40점, 구매는 +50점으로 반영한다.

회사/직무/산업군 점수는 영업 가능성과 제품 적합도를 반영한다. 대기업 또는 1000명 이상 규모는 +15점, 중견 규모는 +10점이다. 직무에 Head, Director, Manager, CTO, CEO가 포함되면 의사결정 가능성이 있다고 보고 +10점을 준다. SaaS, Finance, Healthcare 산업군은 우선순위 산업군으로 보고 +10점을 준다.

캠페인 점수는 유입 경로의 품질을 반영한다. 유효한 캠페인이고 예산이 10000 이상이면 +5점, 채널이 webinar 또는 paid_ads이면 +5점을 준다.

감점 기준도 있다. 캠페인의 UTM 코드가 누락되면 -10점, 최근 이벤트가 전혀 없으면 -10점을 적용한다.

등급은 75점 이상이면 Hot, 45점 이상이면 Warm, 그 외는 Cold다. Manual Review는 점수 구간이 아니라 데이터 품질 기준으로 먼저 분리된다.

추천 액션은 등급에 따라 정한다. Hot은 영업 담당자가 24시간 내 직접 연락하고, Warm은 nurture 이메일 발송 후 3일 내 재확인한다. Cold는 장기 육성 캠페인에 포함하고, Manual Review는 데이터 품질 확인 후 수동 검토한다.

## 10. 검증 결과

현재 코드 기준 실행 결과는 다음과 같다.

- Raw leads count: 12
- Manual Review leads count: 5
- Hot leads count: 2
- Data quality issue count: 12
- pytest: 13 passed

이 결과는 전체 리드 12건 중 7건은 자동 점수 계산 대상이고 5건은 수동 검토 대상이라는 의미다. 수동 검토 대상은 이메일 누락, 중복 이메일, 잘못된 캠페인 참조, 봇성 행동 의심처럼 자동 판단에 사용하기 어려운 문제를 가진 리드다.

Hot Lead는 2건으로, 행동 이벤트와 전환 이벤트, 회사 규모, 직무, 산업군, 캠페인 조건이 함께 높은 리드다. 이 리드는 빠른 영업 후속 대응 대상으로 볼 수 있다.

데이터 품질 이슈는 총 12건이다. 이는 더미데이터에 의도적으로 포함한 품질 문제들이 검증 로직을 통해 탐지되었다는 뜻이다. pytest 13개가 통과한 것은 핵심 검증 규칙, 스코어링 기준, 캠페인 지표 계산의 예외 처리, FastAPI 조회 엔드포인트가 최소한의 테스트로 확인되었다는 의미다.

## 11. 한계와 확장 가능성

이번 구현은 더미데이터 기반이므로 실제 CRM 데이터의 복잡도를 모두 반영하지는 않는다. 실제 운영에서는 데이터 양이 더 많고, 캠페인 구조나 전환 정의도 조직마다 다를 수 있다.

확장 방향으로는 실제 CRM 또는 광고 API 연동이 있다. HubSpot, Salesforce, Google Ads, Meta Ads, 이메일 마케팅 도구에서 데이터를 가져오면 수작업 CSV 준비 과정을 줄일 수 있다.

FastAPI 조회 API를 추가하면 최종 CSV를 직접 열지 않고도 리드 점수와 캠페인 지표를 조회할 수 있다. 대시보드를 연결하면 마케팅과 세일즈 담당자가 Hot Lead, Manual Review, 캠페인별 전환율을 더 쉽게 확인할 수 있다.

이번 선택 구현에서는 FastAPI 조회 API를 단순 조회용으로 추가했다. 핵심 산출물은 여전히 CSV 기반 ETL 파이프라인이며, API는 `data/processed/`에 생성된 결과 파일을 다른 시스템이나 화면에서 조회할 수 있게 하는 확장 기능이다. DB나 인증은 추가하지 않고, 결과 CSV를 읽어 JSON으로 반환하는 범위로 제한했다. 과제 범위를 과도하게 넓히지 않기 위해 대시보드, 인증, 배포 구성은 제외했다.

Airflow 스케줄링을 도입하면 매일 또는 매시간 자동 실행되는 파이프라인으로 확장할 수 있다. 또한 CSV 대신 DB 저장 구조를 사용하면 이력 관리, 증분 처리, 조회 성능 개선이 가능하다.

실제 운영에서는 점수 가중치를 검증해야 한다. 현재 가중치는 설명 가능한 초기 기준이므로, 이후에는 실제 전환율과 영업 결과를 기준으로 A/B 테스트를 수행하고 가중치를 조정하는 과정이 필요하다.

## 12. 결론

이번 MVP는 단순히 리드 점수를 계산하는 데 그치지 않고, 데이터 품질 검증과 리드 우선순위 산정을 하나의 파이프라인으로 구성했다. 원천 데이터 생성부터 정제, 품질 검증, 스코어링, 캠페인 지표 산출, 결과 저장까지 실행 흐름을 분리해 구현했다.

최종 결과는 마케팅과 세일즈 담당자가 다음 액션을 결정할 수 있는 형태로 제공된다. 각 리드는 점수, 등급, 분류 사유, 추천 액션을 가지며, 캠페인별 성과 지표도 함께 산출된다.

이번 구현에서는 설명 가능성과 재현 가능성을 우선했다. 작은 데이터셋에서도 같은 입력으로 같은 결과를 만들 수 있고, 각 판단 기준을 문서와 코드로 확인할 수 있도록 구성했다.
